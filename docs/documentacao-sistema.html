1. Visão Geral do Sistema

Minhas Finanças é uma aplicação web de controle de finanças pessoais desenvolvida com .NET 9 e Blazor WebAssembly. Permite ao usuário registrar receitas, despesas (fixas parceladas e extras), dívidas de terceiros, visualizar dashboard analítico, e exportar relatórios em PDF.

2. Arquitetura da Solução

2.1 Clean Architecture

O projeto adota Clean Architecture (também chamada de Arquitetura Limpa ou Onion Architecture). O princípio central é que as camadas internas não conhecem as camadas externas — a dependência flui sempre de fora para dentro:

🟢 Domain (núcleo)

Entidades do negócio, Value Objects, Enums e Interfaces de Repositório. Não depende de nenhuma outra camada nem de nenhum framework.

MinhasFinancas.Domain

🔵 Application

Casos de uso (Commands e Queries via MediatR), DTOs, Validators. Depende apenas do Domain. Orquestra as regras de negócio.

MinhasFinancas.Application

🟠 Infrastructure

Implementações concretas: EF Core, repositórios, serviços JWT e de senha. Depende do Domain e Application.

MinhasFinancas.Infrastructure

🟣 API (entrada)

Controllers REST, middleware, configuração de DI e pipeline HTTP. É a camada mais externa — depende de tudo.

MinhasFinancas.API
Por que Clean Architecture? Facilita troca de tecnologias (ex.: mudar de PostgreSQL para SQL Server) sem alterar regras de negócio. Garante testabilidade — o Domain e Application podem ser testados sem banco de dados. Promove separação de responsabilidades e torna o código manutenível a longo prazo.

2.3 Fluxo de uma Requisição (Exemplo: criar receita)

  1. Browser (Blazor WASM) — Usuário preenche formulário e clica em Salvar
  2. AuthenticatedHttpHandler — Injeta o JWT Bearer token no cabeçalho Authorization
  3. nginx — Encaminha POST /api/receitas para a API na porta 5090
  4. ErrorHandlingMiddleware — Captura exceções globais antes de chegar no controller
  5. JwtBearerMiddleware — Valida o token JWT; popula HttpContext.User
  6. ReceitasController — Extrai usuarioId do token, cria o Command, envia via ISender.Send()
  7. ValidationBehavior (MediatR Pipeline) — FluentValidation valida os campos antes do handler
  8. CriarReceitaCommandHandler — Chama Receita.Criar(), valida regras de domínio
  9. IReceitaRepository.AdicionarAsync() — Persiste no contexto EF Core
  10. IUnitOfWork.CommitAsync() — Executa SaveChangesAsync() → SQL INSERT no PostgreSQL
  11. Controller — Retorna 201 Created com o ReceitaDto
  12. Blazor — Atualiza a lista de receitas na tela

4. Camada Application (CQRS)

A camada Application implementa os casos de uso do sistema usando o padrão CQRS (Command Query Responsibility Segregation) via biblioteca MediatR. Toda operação de escrita é um Command; toda leitura é uma Query.

4.1 O que são DTOs e por que existem

DTO (Data Transfer Object) — Objeto de Transferência de Dados — é uma classe simples que carrega apenas os dados necessários para a comunicação entre camadas ou entre o sistema e o mundo externo (API ↔ Frontend). Não tem regras de negócio; é só um "envelope" de dados.

Por que não retornar a entidade diretamente?

  • Segurança: A entidade Usuario possui SenhaHash e RefreshToken. O DTO expõe apenas Id, Nome e Email — dados seguros para o frontend.
  • Desacoplamento: A entidade pode mudar internamente sem quebrar o contrato da API.
  • Serialização: Entidades com navegações EF Core (coleções, referências circulares) causam erros de serialização JSON. DTOs são "planos".
  • Performance: O DTO pode expor exatamente o que o cliente precisa, sem campos desnecessários.

4.5 ValidationBehavior (Pipeline MediatR)

public class ValidationBehavior<TRequest, TResponse> : IPipelineBehavior<TRequest, TResponse>
{
    // Executa todos os validators registrados para TRequest
    // Se houver falhas → lança ValidationException (capturada pelo middleware → HTTP 400)
    // Se não houver → chama o próximo handler normalmente
}

Este comportamento elimina a necessidade de validar manualmente em cada controller ou handler — a validação é automática para todo Command/Query.

5.3 Serviços

SenhaService (BCrypt)

MétodoFuncionamento
HashSenha(string)BCrypt com WorkFactor 12 — intencionalmente lento para dificultar brute force
VerificarSenha(string, string)BCrypt.Verify — compara senha plain com hash sem expor o hash
SenhaForte(string)Valida: ≥ 12 chars, tem maiúscula, minúscula, número, caractere especial

TokenService (JWT)

TokenDuraçãoConteúdo
AccessToken (JWT)15 minutosClaims: sub (userId), email, name, jti (UUID único). Assinado com HMAC-SHA256.
RefreshToken7 dias64 bytes aleatórios (CSPRNG) em Base64. Armazenado no banco junto com a expiração.
Fluxo de renovação: Quando o AccessToken expira (15 min), o frontend usa o RefreshToken (armazenado em cookie HttpOnly) para obter um novo par. O RefreshToken do banco é invalidado e substituído por um novo a cada uso — rotação de tokens.

LoginAttemptService (Rate Limiting)

Singleton em memória que registra tentativas de login por e-mail:

  • Após 5 tentativas falhas em 15 minutos → bloqueia o e-mail por 30 minutos
  • Retorna HTTP 429 Too Many Requests com tempo restante de bloqueio
  • Sucesso no login → reseta o contador automaticamente

7.2 State Management com Fluxor (Redux-like)

Por que Fluxor? Em aplicações SPA, múltiplos componentes precisam compartilhar estado (ex.: "usuário está logado?", "qual o nome do usuário?"). O Fluxor implementa o padrão Redux: estado centralizado imutável, actions descrevem intenções, reducers calculam o novo estado, effects lidam com side-effects (chamadas HTTP). 
// Fluxo completo de Login via Fluxor:

// 1. Componente dispara a action
Dispatcher.Dispatch(new LoginAction(email, senha));

// 2. Effect intercepta, chama a API
[EffectMethod]
async Task HandleLogin(LoginAction action, IDispatcher dispatcher)
{
    var response = await authApiService.LoginAsync(action.Email, action.Senha);
    if (response != null)
        dispatcher.Dispatch(new LoginSuccessAction(response.AccessToken, response.NomeUsuario, response.UsuarioId));
    else
        dispatcher.Dispatch(new LoginFailureAction("E-mail ou senha inválidos"));
}

// 3. Reducer atualiza o estado imutavelmente
[ReducerMethod]
static AuthState OnLoginSuccess(AuthState state, LoginSuccessAction action) =>
    state with { IsAuthenticated = true, NomeUsuario = action.NomeUsuario, IsLoading = false };

// 4. Componentes se atualizam automaticamente ao observar o estado

7.3 Autenticação no Frontend

AuthStateProvider implementa AuthenticationStateProvider do ASP.NET e:

  • Lê o JWT do localStorage a cada checagem de estado
  • Decodifica as claims do token (sem validar assinatura — isso é papel da API)
  • Verifica expiração (exp claim)
  • Popula ClaimsPrincipal para que [Authorize] funcione nos componentes Blazor

AuthenticatedHttpHandler é um DelegatingHandler registrado no pipeline do HttpClient:

  • Injeta automaticamente Authorization: Bearer {token} em toda requisição
  • Se receber 401 → dispara logout automático e redireciona para /login

8. Fluxos Principais do Sistema

8.1 Fluxo de Autenticação Completo

  1. Cadastro: POST /api/auth/cadastrar → valida email único + senha forte → hash BCrypt (WF=12) → salva usuário no banco
  2. Login: POST /api/auth/login → verifica rate limit → compara BCrypt → gera JWT (15min) + RefreshToken (7 dias) → seta cookie HttpOnly
  3. Uso da API: Toda requisição carrega Authorization: Bearer {jwt} → middleware valida assinatura, issuer, audience e expiração
  4. Renovação: Quando JWT expira → POST /api/auth/refresh com cookie refreshToken → novo par emitido, token antigo invalidado
  5. Logout: POST /api/auth/logout → remove refreshToken do banco → apaga cookie → frontend limpa localStorage

8.2 Fluxo de Despesa Fixa com Parcelas

  1. Usuário preenche: "Notebook", R$ 3.600, 12 parcelas, março 2026
  2. POST /api/despesas/fixas com os dados
  3. DespesaFixa.Criar() valida as regras de domínio
  4. GerarParcelas(): 3600 ÷ 12 = 300,00 exatos → 12 parcelas de R$ 300,00 (datas: mar/26 a fev/27)
  5. DespesaFixa + Parcelas persistidos em uma transação
  6. No mês de abril, GET /api/despesas/parcelas?mes=4 retorna a parcela 2/12 de R$ 300,00 como pendente
  7. PATCH /api/despesas/parcelas/{id}/pagar marca como paga com a data de pagamento

8.3 Cálculo do Dashboard

  1. GET /api/dashboard?ano=2026&mes=3 (março 2026)
  2. TotalReceitasMes: soma de todas as receitas com DataRecebimento em 03/2026
  3. TotalDespesasMes: soma parcelas vencendo em 03/2026 + despesas extras de 03/2026
  4. SaldoMesAtual: TotalReceitas − TotalDespesas
  5. ProjecaoProximos30Dias: receitas recorrentes − parcelas vencendo nos próximos 30 dias
  6. Evolucao12Meses: array com receitas/despesas/saldo dos últimos 12 meses
  7. DistribuicaoCategorias: agrupamento e percentual por categoria de despesa do mês
